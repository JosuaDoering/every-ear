Why is this warning showing?

What Every Ear just checked

On macOS, Every Ear asks the built-in Application Firewall whether the livekit-server binary is allowed to accept incoming connections. It runs these two checks: 

socketfilterfw --getglobalstate
socketfilterfw --getappblocked <binary>

Right now the firewall is turned on, and for this exact binary it reported “blocking incoming connections”. That is the only reason the warning is here:

What this binary does

livekit-server is the small media server Every Ear runs on this Mac so phones on your network can send and receive the live audio. The Application Firewall decides, per app, whether other devices may open incoming connections to it. When it blocks this binary, those incoming media connections can be dropped.

Why it appears even though audio works right now

The listener and translator phones initiate the connection outward to the server. That outbound handshake usually succeeds even while the app is on the block list, so audio can seem perfectly fine at the moment.

The risk shows up later: when a phone joins from a different subnet, or the network path changes (a Wi-Fi switch, new ICE candidates, a router that needs the server to accept the return path), the blocked incoming rule can suddenly cut the audio. The warning is here so you can fix it before that happens during a live event — not because anything is broken this second.

How to clear it

  1. Click Open Firewall settings below.
  2. Open Firewall Options….
  3. Find livekit-server in the list.
  4. Switch it to Allow incoming connections.
  5. Back in Every Ear, click Re-check on the warning.

Turning the firewall off entirely also clears the warning, but allowing just this one binary is the safer choice.
